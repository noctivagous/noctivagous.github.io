The Terminal Discussed As A Multi-Program Portal

Originally the terminal was the whole computer experience, hence the name 
terminal since it was just a CRT monitor connected to the mainframe 
with a keyboard.  In actuality, how the terminal is used today relies on 
GUI conventions and is placed inside a hybrid setup.  The user will open up one 
or more terminal windows concurrent with other GUI applications.  That of
course wasn't what the CLI was about, but it does work well, like opening
up different virtual computers from the past.  It has a lot of versatility,
often bypasses the friction of the GUI for professional computer work,
and allows certain kinds of work to be done that cannot be done anywhere else.
Every usage of the shell starts at the shell prompt.
From this, two keys type the program to 
list the current directory ls and three keys, ssh, let the user open
a remote shell.  This is a central aspect of its speed, that each new
line of the shell can go in any functional direction by typing out program
names that are often 4 characters or less.

If we are to upgrade the CLI, the terminal, we can identify what still makes it
indispensable, specifally what isn't recognized. Then we can move from that 
to the next generation. First, we would call it a multi-program portal, an 
incidentally timeless interactive scheme because it hosts programs that all 
follow the same rules.  They all have to fit inside the same container. 
They have to be controlled by keyboard in the same way.

The CLI acts as the portal for the programs and this is  
analogous to a multi video game arcade machine that can load different games 
on the same upright cabinet, all of which use the same joystick and buttons. 
The other machines might have customized controls for their games.  
No program can actually go outside of the rules of the terminal 
because the functionality to do it just isn't provided.

Though this interactive scheme was in no way designed to provide these
modern benefits, when programs all have to follow the same appearance,
there is consistency across everything used inside it.  It is like the 
camping multi-tool with all of the utensils that fold out of the 
same holder. 

In contrast with this, every GUI software application launches and manages its 
own user interface and exists separately in multiple aspects, apart from 
emphasizing the graphical analogies of buttons and physical controls.  
Each one will carry wit it its own GUI organization and interactive schemes 
that are particular to its style and purpose.
 
Meanwhile, all terminal apps must behave in the same way,
and if they do provide any kind of persistent GUI the rules are inherited because 
there is no way to do anything but update characters in the display. 
Then there is an important aspect which is that the programs that run inside the 
terminal, if they don't specially exist as persistent GUI, will terminate after 
execution. They are transient by default. With everything driven by text, 
the limited UI for programs links into the ease of typing out commands to 
navigate the file system and manipulate the files with the programs that run 
and then exit. It produces an altogether different dynamic, but since it is old, 
it is often recognized that an upgrade is due.  Everything is keyboard-driven 
and mouse has no role in the CLI. If the mouse is to have a role, it should 
make use of key-clicks where the click of the mouse takes place because a 
keyboard key served as the click button.


The terminal or UNIX shell focuses on the filesystem as its constant reference.  
A program executed is possibly affected by or affects the current working directory.
Some programs in the terminal play a role of commands for this shell even
thought they are necessarily executable programs, such as ls, mv, cp, cd, pwd.  
This distinction isn't made in discussions of UNIX shell, but it is this way 
in practice, with the user typing out a small subset of available programs
most frequently.  A command like "cd" (change directory), when it is 
treated as a command in a next generation of UNIX shell/terminal, can then be mapped 
to a keyboard key or a physical button located to the side of the screen, just to 
give an example.  A conversion of the UNIX terminal to a touchscreen app
might start by placing ls, cd, mv, etc. as buttons to the side of the
text screen (with line break added for "ls").  As soon as you make this
change you then have the opportunity to make a target GUI region for "ls", such
as a tray that slides out.  It then gives you the opportunity to let the user
touch one of the directories or files in that tray.  All of this implies that
the conversion to a different context will lead to and require a lot of
changes, adaptations and revisions.

If there is to be an upgrade to the UNIX shell or terminal it would account for 
classifications of programs' functionality.  In GUI apps, this already takes 
place, where individual functions that would be programs in the UNIX shell are 
made into menu items.

It is worth noting that the required manual use of "ls" adds some occasional
speed to the UNIX shell / terminal because it is always optional for the user
to view the current directory's contents.  Along with this, pressing
the tab key after typing part of a filename means that a destination can be
accessed quickly.

Strictly speaking, calling it "the terminal" is more accurate than 
"command line interface" because it is just an implementation of old terminal
screens for UNIX shells and the interface wasn't exactly the topic at the time. 
We're talking about the terminal because it was the computer interface
and personal computers often were built upon this, which is encoded
characters.  

All of the programs made inside the terminal shell.
