Placing the LLM in a  pre-made component 
environment would give it the structure
it needs to avoid making slop code bases.
But this time there is no need to limit the
range of components to GUI.  It can also 
include customizable building blocks
on the logic end, as well as kits, such as a
prefabricated settings window that
stores its data to a destination.  It might
be called "Basic Settings Kit" and within
the larger system act as a module plugged in.

Premade programmatic elements have usually been 
very low-level for software development, like syncing 
an array of strings with a GUI table 
control.  This notion of them being extended
to the logic side is something different.

An important benefit for the software 
developer is that in a RAD (rapid application
development) environment the output 
is visible right there and can be modified.
It allows the software developer to inspect
what is happening at a slightly higher level.

Sample components that might be bundled
with this LLM-centric RAD editor:

- state machine / state machine manager
- value store / value observer

- REST request and response controller
--- subclass: AI chat API component

When modules are too defined and narrow, 
they become problematic because they are 
inflexible and limited.  If very low level, such as
the notion of an object class, they will
require more management.  The balance
can be genericity, where there are categories
of activities in software development
identified (the use of a state machine)
and the components facilitate that activity
such that they don't need much customization.

The test case might be: what would it take
to produce a versatile publication system
like WordPress.  What premade components, on
the programmatic side and the GUI side
would be provided to the AI?  These are
components and kits that haven't existed before,
both because AI wasn't involved and
software engineers just didn't make them.
It would be the task of the AI to mix, connect,
configure, and sometimes customize these
modules.  Since they are at the code
level, they allow more granular control
than, say, a conventional patch-based 
programming system.  The AI can choose
to subclass or modify these pre-made
components when necessary.

A calculator would be built with premade
logic components that accept OOP messaging.
When the LLM makes the calculator,
it might only require a few instances
of premade components on the logic, data storage,
and presentation sides.

It isn't that the LLM would never write
code, but it would write far less
and thereby stay within guidelines
that prevent excessive code.
The LLM can customize when necessary
and work at a low level for features
and needs specific to the app.

Such a system would integrate more aspects,
such as a design style guide, so that
the LLM follows a framework.
